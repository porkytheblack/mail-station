# Design spec — mailbox-station + gmail-station

This document records the design decisions for the v1 implementation. It complements [mailbox-station.md](./mailbox-station.md) and [gmail-station.md](./gmail-station.md), filling in the specifics those guides leave open.

The corresponding PRD lives at https://github.com/porkytheblack/mail-station/issues/1.

---

## 1. Architecture overview

Two published npm packages plus a conformance test suite, all in a single pnpm workspace:

- `mailbox-station` — provider-agnostic core: outbox commit, trigger worker, Store interface, handler interface.
- `gmail-station` — Gmail provider: API client, OAuth2 refresh, watch lifecycle, Pub/Sub ingress, MIME parser.
- `mailbox-station-conformance` — test battery for verifying user-supplied Store implementations.

Pipeline shape (provider-agnostic):

```
provider emits MailboxEvent
  → core resolves event via MessageResolver (provider-supplied)
  → core commits atomically (insert + cursor + enqueue trigger jobs)
  → core acks the upstream event
  → trigger worker (separate loop): claim → handler → mark done | fail
```

The public API is **Promise + Result-based** for ergonomics. The Effect-typed kernel is exposed at a `/effect` subpath import for users with Effect codebases.

## 2. Workspace and tooling

- pnpm workspaces.
- ESM-only output. Minimum Node 20.
- Build: `tsc` with project references.
- Tests: Vitest.
- Versioning: Changesets.
- Each package's `exports` map exposes a default entry and an `/effect` subpath. Internal modules live under `internal/` and are not re-exported.

## 3. Public API

### 3.1 `createStation` factory

```ts
const station = createStation({
  store,                  // StoreAdapter
  handler,                // MessageHandlerFn
  config: WorkerConfig,
  providers: {            // typed by keys
    gmail: gmailProvider({ ... }),
  },
  logger?: StationLogger,
})

await station.start()                       // resolves once long-runners are up
await station.providers.gmail.register({ userId, emailAddress, refreshToken })
await station.providers.gmail.renewExpiringWatches()
process.on("SIGTERM", () => station.stop()) // graceful shutdown
await station.wait()                        // resolves when stopped
```

Three lifecycle verbs: `start`, `wait`, `stop`. The providers map is typed by the keys passed in — `station.providers.gmail` exists in TypeScript only if `gmail` is a key in the input map.

### 3.2 Effect-typed API

```ts
import { createStationEffect } from "@your-org/mailbox-station/effect"
```

Same kernel, layer-based skin. For users who already use Effect-TS and want native composition.

### 3.3 Result type

```ts
type Result<T, E> =
  | { ok: true;  value: T }
  | { ok: false; error: E }

const ok  = <T>(value: T): Result<T, never> => ({ ok: true, value })
const err = <E>(error: E): Result<never, E> => ({ ok: false, error })
```

Adapter and handler return `Promise<Result<T, E>>`. Uncaught throws are caught by the kernel and treated as `{ _tag: "Transient" }` — backstop for buggy implementations, not the primary path.

## 4. Data model

### 4.1 `MailMessage`

```ts
type MailMessage = {
  messageId:    MessageId        // provider-native id, scoped by (accountId, messageId)
  threadId:     ThreadId | null
  accountId:    AccountId
  provider:     "gmail" | "outlook"

  from:         EmailAddress
  to:           EmailAddress[]
  cc:           EmailAddress[]
  bcc:          EmailAddress[]
  replyTo:      EmailAddress[]
  subject:      string

  bodyText:     string           // both can be present, both default to ""
  bodyHtml:     string

  headers:      Record<string, string[]>   // lowercased keys, multi-value
  attachments:  AttachmentRef[]
  labels:       string[]         // raw provider strings, NOT normalized

  receivedAt:   Date
  sentAt:       Date | null
  sizeEstimate: number
}

type AttachmentRef = {
  attachmentId: string
  filename:     string
  mimeType:     string
  sizeBytes:    number
  contentId:    string | null
  inline:       boolean
}

type EmailAddress = { name: string | null; email: string }
```

**Decisions:**
- Labels stay raw. Gmail's `INBOX` and Outlook's `Inbox` are different concepts (multi-assignment vs single-parent); a normalized field would lie about the model.
- Headers are multi-value. RFC 5322 allows duplicate headers (`Received`, `Authentication-Results`); flattening drops information.
- Attachments are refs only. Bytes fetched via a separate Gmail-client call when needed.
- `messageId` is provider-native (Gmail's `id`), scoped uniquely by `(accountId, messageId)`.
- Charset assumed UTF-8 in v1 (legacy non-UTF-8 mail decodes incorrectly).

### 4.2 `MailboxAccount`

```ts
type MailboxAccount = {
  accountId:       AccountId      // generated by core (crypto.randomUUID)
  userId:          UserId         // opaque, consumer-supplied
  provider:        "gmail" | "outlook"   // closed union
  emailAddress:    string         // lowercased
  status:          "active" | "paused" | "revoked"
  credentials:     Record<string, unknown>  // opaque to core
  lastEventCursor: string | null
  watchExpiresAt:  Date | null
  createdAt:       Date
  updatedAt:       Date
}
```

Unique key: `(provider, emailAddress)`. Cross-user duplicates rejected. Encryption at rest is the adapter's concern.

### 4.3 `MailboxEvent`

```ts
type MailboxEvent = {
  eventId:         string
  providerPayload: unknown        // opaque to core; resolver decodes
  receivedAt:      Date
}
```

Provider-specific payload shape (e.g., Gmail's `{ emailAddress, historyId }`) is declared as an Effect Schema in the provider package and validated at the resolver boundary.

### 4.4 `TriggerJob`

```ts
type TriggerJob = {
  jobId:           JobId
  messageId:       MessageId
  accountId:       AccountId
  state:           "pending" | "succeeded" | "failed"
  attempts:        number
  lastError:       string | null
  nextAttemptAt:   Date | null    // null when state != "pending"
  claimedAt:       Date | null
  claimedBy:       string | null
  leaseExpiresAt:  Date | null
  createdAt:       Date
  completedAt:     Date | null    // set on terminal transition
}
```

Explicit state machine. Successful and dead-letter rows are kept for observability — the adapter decides on pruning policy.

## 5. Store contract

The Store interface — 9 Promise<Result<T, E>> methods. Adapter author implements one for their backend (Postgres, SQLite, Redis, HTTP API, etc.).

### 5.1 Methods

```ts
type StoreAdapter = {
  // Account lifecycle
  createAccount:             (input: CreateAccountInput) => Promise<Result<MailboxAccount, StoreError>>
  getAccount:                (accountId: AccountId) => Promise<Result<MailboxAccount, StoreError>>
  getAccountByEmail:         (provider: Provider, email: string) => Promise<Result<MailboxAccount, StoreError>>
  updateAccount:             (accountId: AccountId, patch: AccountPatch) => Promise<Result<MailboxAccount, StoreError>>
  listAccountsExpiringWatch: (provider: Provider, before: Date) => Promise<Result<MailboxAccount[], StoreError>>

  // Atomic commit (the heart of the outbox)
  commitMessages:            (input: CommitMessagesInput) => Promise<Result<{ committedMessageIds: MessageId[] }, StoreError>>

  // Trigger outbox
  claimTriggerJobs:          (input: ClaimTriggerInput) => Promise<Result<ClaimedJob[], StoreError>>
  markTriggerDone:           (jobId: JobId) => Promise<Result<void, StoreError>>
  markTriggerFailed:         (jobId: JobId, error: string, nextAttemptAt: Date | null) => Promise<Result<void, StoreError>>
}

type StoreError =
  | { _tag: "Transient";        message: string; cause?: unknown }
  | { _tag: "Permanent";        message: string; cause?: unknown }
  | { _tag: "AccountNotFound";  accountId?: AccountId; emailAddress?: string }
  | { _tag: "DuplicateAccount"; provider: Provider; emailAddress: string }

type ClaimedJob = { job: TriggerJob; message: MailMessage }   // joined; one round trip
```

### 5.2 Invariants

- **`commitMessages` is one atomic operation.** Insert messages (idempotent on `(accountId, messageId)`), advance cursor, enqueue trigger jobs for newly-inserted messages — all-or-nothing.
- **Idempotency lives in the adapter.** Postgres: `INSERT ... ON CONFLICT DO NOTHING`; SQLite: `INSERT OR IGNORE`. The conformance suite asserts this.
- **`claimTriggerJobs` is mutex-safe under concurrent calls.** Postgres: `SELECT ... FOR UPDATE SKIP LOCKED`; SQLite: `BEGIN IMMEDIATE` + UPDATE.
- **Lease-based claiming.** Worker sets `claimedAt = now`, `leaseExpiresAt = now + leaseDurationMs`. Crashed-worker recovery: another worker can claim once `leaseExpiresAt < now`.
- **`markTriggerDone` keeps the row** with `state = succeeded`. **`markTriggerFailed(_, _, null)` is terminal** with `state = failed`. Both stay in storage for observability.
- **`now` is passed in, not read from the Store's clock.** Lets `TestClock` control time deterministically.

### 5.3 No methods in v1

- No `getMessage` (worker gets it via `claimTriggerJobs`).
- No `deleteAccount` / `deleteMessage` (out of scope).
- No batch `commitMessages` across accounts (single-account scope per call).

## 6. Pipeline (ingress↔core handshake)

**Pattern: return-value.** Ingress calls `pipeline.processEvent(event)` and gets back `"ack" | "nack"`. Ingress applies the decision to the upstream messaging primitive (Pub/Sub message ack/nack).

### 6.1 Core method

```ts
interface MailboxPipeline {
  processEvent: (event: MailboxEvent) => Promise<"ack" | "nack">
}
```

Internally: resolver → commitMessages → ack/nack decision based on error tag (see section 8.2).

### 6.2 `MessageResolver` contract

```ts
interface MessageResolver {
  resolve: (event: MailboxEvent) => Promise<Result<ResolveResult, ResolverError>>
}

type ResolveResult = {
  accountId: AccountId
  messages:  MailMessage[]    // may be empty (label-only history)
  newCursor: string           // advanced even when messages is empty
}
```

- Empty messages array still advances the cursor.
- `HistoryGoneError` is **swallowed inside the resolver** — cursor advances to the notification's historyId, returns empty messages. Never surfaces to pipeline.
- `messages.get` partial failure: 404 skipped (deleted between list-and-get); all other errors abort the whole event.

### 6.3 `MessageHandler` contract

```ts
type MessageHandlerFn = (
  message: MailMessage,
  ctx: HandlerContext,
) => Promise<Result<void, HandlerError>>

type HandlerContext = {
  jobId:     JobId
  accountId: AccountId
  attempt:   number   // 1-based; 1 on first run
}

type HandlerError =
  | { _tag: "Transient"; message: string; cause?: unknown }   // retry until maxAttempts
  | { _tag: "Permanent"; message: string; cause?: unknown }   // dead-letter immediately
```

`HandlerContext` deliberately omits internal outbox fields (lastError, claimedBy, leaseExpiresAt) — those are debug, not behavior.

## 7. Trigger worker

### 7.1 Worker config defaults

```ts
type WorkerConfig = {
  workerId:           string  // default: `${hostname}-${pid}-${shortRandom}`
  triggerConcurrency: number  // default 8
  claimBatchSize:     number  // default 16  (≥ concurrency)
  leaseDurationMs:    number  // default 5 * 60 * 1000   (5 min)
  idlePollIntervalMs: number  // default 1000
  maxAttempts:        number  // default 10
  backoff: {
    baseMs:           number  // default 30 * 1000        (30 s)
    factor:           number  // default 2
    maxMs:            number  // default 5 * 60 * 1000    (5 min)
    jitterFactor:     number  // default 0.25 (±25%)
  }
}
```

### 7.2 Backoff schedule

With defaults: `30s, 1m, 2m, 4m, 5m, 5m, 5m, 5m, 5m, 5m` — capped at 5 min.

Total span before terminal: ~40 minutes. If still failing at 5-min intervals after several tries, it's a real bug, not a transient hiccup.

### 7.3 Loop pseudocode

```
forever:
  claimed = claimTriggerJobs({ workerId, limit: claimBatchSize, lease, now })
  if claimed.empty:
    sleep(idlePollIntervalMs); continue
  forEach(claimed, runOne, { concurrency: triggerConcurrency })

runOne(claimed):
  attempt = claimed.job.attempts + 1
  result = handler(claimed.message, { jobId, accountId, attempt })
  if result.ok:
    markTriggerDone(jobId)
  else if attempt >= maxAttempts:
    markTriggerFailed(jobId, errStr, nextAttemptAt: null)   // terminal
  else:
    delay = min(maxMs, baseMs * factor^attempts) * jitter(0.75..1.25)
    markTriggerFailed(jobId, errStr, nextAttemptAt: now + delay)
```

### 7.4 Decisions

- **`claimBatchSize >= triggerConcurrency`.** Default 16 vs 8 keeps the pipeline full while in-flight handlers complete. Equal would stall.
- **No lease heartbeating in v1.** Configurable lease only. If user has long handlers, they bump `leaseDurationMs`. Heartbeat is v2.
- **No per-account serialization.** Two messages for the same account run concurrently. Per-key concurrency is v2.
- **WorkerId default includes random suffix.** Hostname+pid alone collides under fork models or container restarts.

## 8. Errors and observability

### 8.1 Error catalog

```ts
type ResolverError =
  | { _tag: "MalformedNotification"; details: string }
  | { _tag: "AccountNotFound";       emailAddress: string }
  | { _tag: "AccountPaused";         accountId: AccountId }
  | { _tag: "AccountRevoked";        accountId: AccountId }
  | { _tag: "CredentialsRevoked";    accountId: AccountId; reason: string }
  | { _tag: "ProviderTransient";     message: string; statusCode?: number; cause?: unknown }
  | { _tag: "ProviderPermanent";     message: string; statusCode?: number; cause?: unknown }

type IngressError =
  | { _tag: "DecodeError";           message: string }
  | { _tag: "SubscriptionError";     message: string; cause?: unknown }

// StoreError defined in section 5.1
```

`HistoryGone` is internally handled by the Gmail resolver — it never surfaces to the pipeline.

### 8.2 Pipeline action mapping

| Error                              | Upstream ack/nack | Account state                  | Log level |
| ---------------------------------- | ----------------- | ------------------------------ | --------- |
| Resolver: MalformedNotification    | ack               | none                           | warn      |
| Resolver: AccountNotFound          | ack               | none                           | info      |
| Resolver: AccountPaused            | ack               | none                           | debug     |
| Resolver: AccountRevoked           | ack               | none (already revoked)         | debug     |
| Resolver: CredentialsRevoked       | ack               | **mark `status: "revoked"`**   | error     |
| Resolver: ProviderTransient        | nack              | none                           | warn      |
| Resolver: ProviderPermanent        | ack               | none                           | error     |
| Store: Transient (during commit)   | nack              | none                           | warn      |
| Store: Permanent (during commit)   | ack + alarm       | none                           | error     |
| Ingress: DecodeError               | ack               | none                           | warn      |
| Ingress: SubscriptionError         | (loop dies)       | none                           | error     |

**Notes:**
- `CredentialsRevoked` is the only error that mutates account state.
- `Store.Permanent` acks (not nacks) to prevent infinite redelivery against a broken DB. High-severity log event must be alarm-routed.
- `SubscriptionError` kills the ingress fiber; supervised restart with exponential backoff (1s, 5s, 30s, capped) up to 10 attempts.

### 8.3 Observability

```ts
type StationLogger = {
  debug: (event: string, fields?: Record<string, unknown>) => void
  info:  (event: string, fields?: Record<string, unknown>) => void
  warn:  (event: string, fields?: Record<string, unknown>) => void
  error: (event: string, fields?: Record<string, unknown>) => void
}
```

**Stable event names (part of the public API contract):**

```
account.registered          { accountId, provider, emailAddress }
account.revoked             { accountId, reason }
account.watch_renewed       { accountId, oldExpiry, newExpiry }
event.received              { eventId }
event.committed             { eventId, accountId, messagesInserted, cursorAdvanced }
event.dropped               { eventId, reason, errorTag }
event.history_gone          { accountId, oldCursor }
message.gone_during_fetch   { messageId, accountId }
trigger.claimed             { jobId, messageId, accountId, attempts }
trigger.succeeded           { jobId, durationMs }
trigger.failed              { jobId, error, attempts, nextAttemptAt }
trigger.dead_lettered       { jobId, error, attempts }
ingress.started             { provider, subscription }
ingress.subscription_lost   { provider, error }
```

Renaming events is a breaking change. Adding new events is non-breaking.

**No state-change push callbacks in v1.** Consumer queries Store for `WHERE status = 'revoked'`. **No metrics SDK.** Logger events only; consumer routes to their backend.

## 9. Gmail provider (`gmail-station`)

### 9.1 Configuration

```ts
type GmailConfig = {
  googleClientId:     string
  googleClientSecret: string
  gcpProjectId:       string
  pubsubTopic:        string                          // for users.watch
  pubsubSubscription: string                          // for ingress pull
  labelFilter:        readonly string[] | null        // ["INBOX"] default; null = watch all
  pullConcurrency:    number                          // default 4
  renewalWindowMs:    number                          // default 24 * 60 * 60 * 1000 (24h)
  pubsubAuth?:
    | { kind: "adc" }                                 // default
    | { kind: "keyFile";     keyFilename: string }
    | { kind: "credentials"; credentials: ServiceAccountKey }
}

type GmailCredentials = {     // stored opaquely in MailboxAccount.credentials
  refreshToken:          string
  accessToken?:          string
  accessTokenExpiresAt?: Date
}
```

### 9.2 Gmail client + OAuth2

- **Library:** `@googleapis/gmail` (split per-API package) + `google-auth-library`. Avoids the monolithic `googleapis` weight.
- **Per-call `OAuth2Client` construction.** Each request constructs a fresh client from the account's credentials. Subscribes to the `tokens` event for fire-and-forget writeback to Store. No pooling.
- **Request-level retry:** 3 attempts, exponential backoff (200ms, 1s, 5s) for 5xx and network errors. 4xx other than 429 surfaces immediately as `ProviderPermanent`.
- **`messages.get?format=full`** always — gives the parsed MIME tree.
- **`invalid_grant` detection** is the single signal for `CredentialsRevoked`.

### 9.3 Watch lifecycle

**`register({ userId, emailAddress, refreshToken })` flow:**

1. Validate refresh token via token-endpoint refresh. Fast-fail on `invalid_grant`.
2. Pre-flight `getAccountByEmail` check. Avoids wasted `users.watch` on duplicates.
3. Call `users.watch({ topicName, labelIds, labelFilterBehavior: "INCLUDE" })`.
4. `Store.createAccount` with credentials, cursor, watch expiration.
5. On persist failure: best-effort `users.stop` (compensating); 7-day natural expiration as backstop.

**`renewExpiringWatches()` flow:**

1. `Store.listAccountsExpiringWatch("gmail", now + renewalWindowMs)`.
2. Parallel renew with concurrency 8.
3. Per account: `users.watch` → `Store.updateAccount` with new cursor and expiration.
4. `invalid_grant` during renewal → flip account to `status = revoked`.
5. Aggregate result returned: `{ renewed, failed, details }`. Per-account failures don't fail the cron.

**Pub/Sub topic + subscription must be pre-existing.** Package does not provision GCP resources. Documentation directs user to `gcloud`/Terraform.

### 9.4 Pub/Sub ingress

- **Library:** `@google-cloud/pubsub` (official). gRPC streaming-pull is too thorny to hand-roll.
- **Single subscription consumes all watched accounts' notifications.** Per-account subscriptions infeasible at scale. Routing happens in resolver via `getAccountByEmail`.
- **Auth:** ADC default. Explicit `keyFile` or `credentials` config as escape hatch.
- **`flowControl.maxMessages = pullConcurrency`** — backpressure.
- **Initial `ackDeadline: 60s`**, SDK auto-extends up to 600s.
- **Startup assertions:** subscription exists AND is pull-type. Fail fast with clear errors.
- **Per-message handler in independent fiber.** Uncaught throws ack and emit log event (don't kill the loop).
- **`subscription.on("error")` kills the fiber.** Supervised restart with exponential backoff.

### 9.5 Resolver (history.list + messages.get)

Flow per `MailboxEvent`:

1. Decode payload via Effect Schema → `{ emailAddress, historyId }`.
2. `getAccountByEmail("gmail", emailAddress)`. Short-circuit on paused/revoked.
3. Build per-account Gmail client.
4. Iterate ALL `history.list` pages with `historyTypes: ["messageAdded"]`, `maxResults: 500`. Server-side `labelId` filter only when single-label config; else client-side filter from `message.labelIds`.
5. Save `historyId` from the LAST page's response as the new cursor.
6. Dedup messageIds across history entries.
7. Fetch messages in parallel (concurrency 8). Each fetch parses MIME → `MailMessage`.
8. Partial-failure policy: 404 skipped (deleted between list and get); all other failures abort whole event (idempotency on commit makes redelivery cheap).
9. Return `{ accountId, messages, newCursor }`.

**Special case:** `history.list` 404 (`HistoryGone`) → cursor advances to notification's historyId, returns empty messages. Logged as `event.history_gone`.

### 9.6 MIME parser

- **Hand-rolled tree walker** over Gmail's `payload.parts`. Doesn't re-parse RFC822 — Gmail already structured the tree.
- `addressparser` library for RFC5322 address fields.
- `Buffer.from(s, "base64url")` directly for body decoding (Node 16+ native).
- **First text/plain wins for bodyText; first text/html wins for bodyHtml; depth-first.**
- **Attachments:** anything with `Content-Disposition: attachment` or non-empty `filename`. **Inline images:** `Content-Disposition: inline` + Content-ID → attachment with `inline: true`.
- **`message/rfc822` parts (forwards) NOT recursed.** Treated as opaque.
- **Charset = always UTF-8** in v1. Documented limitation.
- **No HTML sanitization.** `bodyHtml` is raw; consumer sanitizes for their render context.
- **Empty/missing fields tolerated** — empty strings, null timestamps, empty arrays. Parser never throws.

## 12. Conformance suite

Adapter author installs `@your-org/mailbox-station-conformance` as a dev dep and calls:

```ts
import { runStoreConformance } from "@your-org/mailbox-station-conformance"

runStoreConformance({
  name: "my-sqlite-store",
  makeStore: async () => ({
    store:    await createMySqliteStore({ url: ":memory:" }),
    teardown: async () => { /* close db */ },
  }),
})
```

Internally calls Vitest `describe`/`it`. Fresh Store per test (default isolation).

**Battery (~25 tests):**

```
account lifecycle
  ✓ createAccount returns a new account with generated accountId
  ✓ createAccount with existing (provider, email) → DuplicateAccount
  ✓ getAccount nonexistent → AccountNotFound
  ✓ getAccountByEmail nonexistent → AccountNotFound
  ✓ updateAccount changes credentials, status, cursor, watchExpiresAt independently
  ✓ listAccountsExpiringWatch filters by provider AND watchExpiresAt < cutoff

commitMessages atomicity & idempotency
  ✓ inserts new messages, advances cursor, enqueues 1 trigger job per new message
  ✓ second call with same messages: no duplicate inserts, cursor still advances
  ✓ committedMessageIds reflects only newly-inserted (empty on full re-commit)
  ✓ partial overlap: 3 new + 2 existing → 3 trigger jobs, 3 in committedMessageIds
  ✓ commitMessages with empty messages array still advances cursor

trigger outbox
  ✓ claimTriggerJobs respects nextAttemptAt > now (not yet ready)
  ✓ claimTriggerJobs respects state ≠ "pending"
  ✓ claimTriggerJobs respects valid lease
  ✓ claimTriggerJobs returns expired-lease jobs as claimable
  ✓ concurrent claimTriggerJobs from two workers: no overlap (mutex invariant)
  ✓ claimTriggerJobs respects limit
  ✓ claimTriggerJobs returns ClaimedJob with full message joined

trigger state transitions
  ✓ markTriggerDone: state=succeeded, completedAt set, no longer claimable
  ✓ markTriggerFailed with future date: state stays "pending", attempts+=1
  ✓ markTriggerFailed with null: state="failed", completedAt set, no longer claimable
```

A **meta-test** runs the full suite against an in-memory reference Store (shipped in the conformance package) — proves the suite catches violations and the reference impl is correct.

## 13. Module breakdown

### `mailbox-station` internals

## 15. Operational considerations

- **`Store.Permanent` during commit acks (not nacks)** to prevent redelivery storms against a broken DB. High-severity log event must be alarm-routed by the operator.
- **Watch leak risk on `register`** (watch live, persist fails) is mitigated by best-effort `users.stop` and 7-day natural watch expiration as backstop.
- **Renewal cron cadence is the operator's responsibility.** Default 24-hour renewal window assumes daily cron; weekly cron operators must increase the window.
- **Worker process scaling is via running additional processes** against the same Store. Lease-based claims ensure at-most-one-in-flight per job across processes.
- **Multi-label `users.watch` falls back to client-side filtering** in the resolver due to a Gmail API limitation (history.list takes a single labelId).
- **`SubscriptionError` supervised restart** uses exponential backoff up to 10 attempts, then gives up. Repeated restarts mean Pub/Sub credentials or subscription config is bad.

## 16. Public API stability commitments

- Stable log event names are part of the public contract; renaming is breaking.
- The `Result` error tag union for Store and handler is part of the public contract.
- Adding new error tags is non-breaking for consumers using exhaustive switches with a default branch.
- The `/effect` subpath API uses the same kernel as the default Promise API; behavior parity is property-tested in CI.

## 17. Dependency footprint

- **Core:** zero runtime dependencies aside from `effect`.
- **Gmail provider:** `@googleapis/gmail`, `google-auth-library`, `@google-cloud/pubsub`, `addressparser`.
- **Conformance:** `vitest` (peer) plus an in-memory reference Store implementation for meta-testing.
